Mapa de conceitos

o que você vai dominar em cada frente
📒 Contábil
Partidas dobradas e natureza das contas
Todo lançamento contábil tem dois lados de igual valor: débito e crédito. A natureza da conta define qual lado aumenta o saldo — ativo aumenta com débito, passivo e receita aumentam com crédito.
Plano de contas e lançamentos
O plano de contas é a lista numerada de todas as contas que o negócio usa (ex: 1.1.1 Caixa, 2.1.1 Fornecedores). Um lançamento registra uma operação informando quais contas foram debitadas e creditadas.
DRE, Balanço Patrimonial, DFC
A DRE mostra se o negócio lucrou em um período. O Balanço mostra o que ele tem (ativos) e o que deve (passivos) em uma data. A DFC mostra quanto dinheiro entrou e saiu — diferente do lucro, que inclui vendas ainda não recebidas.
Auditoria e controles internos
Auditoria é verificar se os lançamentos são verdadeiros e corretos. Controles internos são regras que previnem erros antes que aconteçam — como exigir aprovação dupla em ajustes de estoque acima de certo valor.
Contabilidade de custos e margem
Custo de um produto inclui compra, frete, seguro e perdas — não só o preço de compra. Margem bruta é Receita menos CMV (custo das mercadorias vendidas). Margem de contribuição desconta também os custos variáveis.
Tributário aplicado por operação
Cada operação de estoque pode gerar obrigações fiscais diferentes: entrada gera crédito de ICMS, saída gera débito de ICMS + PIS/COFINS. O regime tributário da empresa (Simples, Lucro Presumido, Real) muda os cálculos.
Fechamento mensal e reconciliação
Fechamento é "encerrar os livros" do mês: verificar que todos os lançamentos estão corretos e gerar as demonstrações. Reconciliação é cruzar o saldo do sistema com o físico e explicar qualquer diferença encontrada.
📊 Estatística e IA
Tendência, sazonalidade e ruído
Tendência é a direção geral da série (subindo, descendo). Sazonalidade é um padrão que se repete em ciclos fixos (mais vendas em dezembro). Ruído é variação aleatória sem padrão — nenhum modelo consegue prever perfeitamente.
Stationarity, ACF/PACF e ARIMA/SARIMA
Série estacionária é aquela sem tendência ou sazonalidade — exigida pelo ARIMA. ACF e PACF são gráficos que revelam os parâmetros p e q do modelo. SARIMA adiciona componente sazonal ao ARIMA.
Métricas: MAE, RMSE, MAPE, WAPE
MAE é o erro médio absoluto. RMSE penaliza erros grandes. MAPE é o erro em percentual — fácil de explicar ao cliente. WAPE é o MAPE ponderado por volume, mais justo quando SKUs têm volumes muito diferentes.
Janela temporal, horizonte e lookback
Lookback (janela temporal) é quantos períodos passados o modelo usa para prever. Horizonte é quantos períodos à frente ele prevê. Uma janela de 12 meses e horizonte de 4 semanas são escolhas típicas para reposição de estoque.
Features exógenas (feriados, clima, eventos)
Variáveis externas que influenciam a demanda e não estão na própria série histórica. O modelo aprende que vendas caem na semana do carnaval ou sobem em Black Friday. Sem elas, o modelo acha que são variações aleatórias.
Overfitting, drift e recalibração
Overfitting: modelo que decorou o treino mas erra no mundo real. Drift: o padrão do mundo muda e o modelo envelhece. Recalibração: retreinar o modelo periodicamente com dados recentes para manter a precisão.
MLOps: versionamento e rollback de modelos
MLOps é o Git para modelos de ML: rastreia qual versão rodou em produção, com quais dados foi treinada e quais métricas tinha. Rollback é voltar para a versão anterior quando o modelo atual começa a errar mais.
🔐 API e Operação
Semântica HTTP e status codes com propósito
200 OK, 201 Created, 400 Bad Request, 409 Conflict, 422 Unprocessable, 503 Unavailable — cada código tem um significado específico. Usar o código errado confunde o cliente da API e dificulta o diagnóstico de erros.
Idempotência, versionamento e deprecação
Idempotência: chamar o mesmo endpoint duas vezes produz o mesmo resultado que chamar uma vez. Versionamento: /v1/ e /v2/ coexistem durante a transição. Deprecação: avisar com antecedência antes de remover um endpoint.
Transactional Outbox e DLQ
Outbox: salvar o evento no banco junto com o dado na mesma transação — garantindo que nunca se perca na publicação. DLQ (Dead-Letter Queue): fila de destino para eventos que falharam muitas vezes, sem descarte silencioso.
OAuth2, OIDC e PKCE
OAuth2 é o protocolo de autorização ("o que você pode fazer"). OIDC adiciona identidade ("quem você é") — é o que o botão "Entrar com Google" usa. PKCE é uma extensão de segurança obrigatória para apps mobile e desktop.
Roles, escopos e permissões por tenant
Role é um perfil de acesso (operador, gerente, contador). Scope é uma permissão específica (wms:write, contabil:read). Tenant é o isolamento de dados por cliente — o operador da Empresa A nunca vê dados da Empresa B.
Logs estruturados e tracing por correlation_id
Log estruturado é JSON com campos fixos (level, timestamp, correlation_id, tenant_id) — não texto livre. Tracing com correlation_id permite rastrear uma operação que passou por WMS → Event Store → Contábil em segundos.
SLOs, alertas e runbooks
SLO (Service Level Objective) é a meta de qualidade: "99.5% das requisições em menos de 500ms". Alerta acionável diz o que fazer, não só que algo está errado. Runbook é o passo a passo de como resolver cada tipo de incidente.

API madura em produção

Contrato externo estável, semântica HTTP, idempotência e versionamento
Prioridade 1 — Produto
🧠 Por que isso importa para o Jade-stock

O WMS já tem API interna. Mas uma API pública — consumida pelo SDK, por parceiros e pelo cliente rico — precisa de regras mais rígidas: status codes que fazem sentido, erros com correlation_id para rastrear, versão explícita para não quebrar clientes antigos. Sem isso, cada mudança no backend vira uma dor de cabeça no front-end.

O que dominar

  • Semântica HTTP completa: quando usar 200, 201, 400, 409, 422, 503 — e por quê cada um existe.
  • Padrão de erro com error_code, message e correlation_id em todos os endpoints.
  • Idempotência via chave no header (Idempotency-Key) para operações críticas como pagamentos e lançamentos.
  • Versionamento de API: path (/v1/) vs header. Como deprecar uma versão sem quebrar clientes.
  • Paginação com cursor vs offset — e quando usar cada um.

Aplicação no Jade-stock

  • Padrão único de erro/retorno em todos endpoints v1 — hoje cada endpoint retorna diferente.
  • SDK alinhado ao OpenAPI publicado: front-end nunca constrói request na mão.
  • Paginação em endpoints que retornam listas de movimentações ou eventos históricos.
  • Contrato de integração explícito para quando um parceiro consumir a API do Jade-stock.

PostgreSQL — concorrência e consistência

Travas transacionais para não corromper saldo nem duplicar eventos
Prioridade 1 — Produto
🧠 Por que isso importa para o Jade-stock

O WMS já tem o teste test_concorrencia_transferencia_sem_corrupcao_de_saldo passando — o que significa que você já lida com concorrência. Mas você precisa entender por que aquele código funciona: qual isolation level está sendo usado, o que acontece se dois usuários tentam dar baixa no mesmo SKU ao mesmo tempo, e como o PostgreSQL resolve isso sem que você precise implementar nada manual.

O que dominar

  • Isolation levels: Read Committed (padrão), Repeatable Read e Serializable — o que muda em cada um.
  • SELECT FOR UPDATE: como travar uma linha antes de modificar, evitando race condition de saldo.
  • Deadlock: o que é, como detectar no log do Postgres e como evitar pela ordem de atualização.
  • JSONB: como armazenar payloads de eventos com schema flexível e consultar eficientemente.
  • Índices compostos: quando criar, qual ordem de colunas importa e como verificar se estão sendo usados.

Aplicação no Jade-stock

  • Atomicidade: saldo + movimentação + evento na mesma transação — já implementado, agora você entende o porquê.
  • Índice em (tenant_id, occurred_at DESC) na event_store.events para consultas rápidas.
  • Consultas de analytics (relatórios de IA) usando replicas de leitura para não travar o operacional.
  • JSONB no campo payload dos eventos — com índice GIN para busca dentro do JSON.

Arquitetura orientada a eventos

Fundação para IA e Contábil consumirem o WMS sem acoplamento frágil
Prioridade 1 — Produto
🧠 Por que isso importa para o Jade-stock

A Event Store já existe na arquitetura do Jade-stock. O que você precisa entender agora é como garantir que um evento nunca se perca entre o banco e o consumidor (Outbox Pattern), como o módulo Contábil pode ser reiniciado sem processar eventos duas vezes (Idempotent Consumer), e o que acontece quando um evento falha repetidamente (Dead-Letter Queue).

O que dominar

  • Transactional Outbox: por que salvar o evento no banco na mesma transação que o dado — e só depois publicar.
  • Idempotent Consumer: como garantir que processar o mesmo evento duas vezes não duplica o lançamento contábil.
  • Retry com backoff exponencial: esperar 30s, 2min, 10min antes de desistir.
  • Dead-Letter Queue (DLQ): onde vai o evento que falhou 5 vezes — e como reprocessar com segurança.
  • Versionamento de evento: como adicionar campos ao payload sem quebrar consumidores antigos.

Aplicação no Jade-stock

  • WMS publica fatos imutáveis na event_store.events — nunca modifica, só insere.
  • Contábil consome e lança sem tocar no saldo do WMS — separação total por schema.
  • IA consome histórico de movimentações para treinar modelos de previsão.
  • Evento com status = 'dead_letter' nunca é silenciosamente descartado — fica disponível para reprocessamento manual.

Estatística aplicada — ARIMA e SARIMA

Base obrigatória antes de rede neural: previsão explicável e benchmark honesto
Prioridade 2 — Dados / IA
🧠 Por que isso importa para o Jade-stock

Antes de implementar qualquer modelo de IA para prever demanda de estoque, você precisa de um baseline honesto. O ARIMA é esse baseline: simples, explicável, auditável. Se o modelo neural não vencer o ARIMA com margem consistente, não vai para produção. Você precisa entender ARIMA para saber quando a IA está realmente ajudando — ou só adicionando complexidade.

O que dominar

  • Tendência (trend): série que sobe ou desce ao longo do tempo.
  • Sazonalidade (seasonality): padrão que se repete em ciclos (semanal, mensal, anual).
  • Ruído (noise): variação aleatória que não tem padrão — e que modelos simples não conseguem prever.
  • Stationarity: série "estável" que ARIMA exige. Como testar (Teste ADF) e como transformar (diferenciação).
  • ACF e PACF: gráficos que revelam os parâmetros p, d, q do ARIMA. Leia os gráficos antes de ajustar na força bruta.
  • Métricas de avaliação: MAE (erro médio), RMSE (penaliza erros grandes), MAPE (erro percentual), WAPE (ponderado por volume).
  • SARIMA: ARIMA com componente sazonal. Quando usar em vez do ARIMA simples.

Aplicação no Jade-stock

  • Baseline por SKU/endereço/período: o modelo mais simples possível para comparação futura.
  • Comparar baseline ingênuo (média dos últimos 4 períodos) vs ARIMA — e documentar o ganho.
  • Publicar recomendação, nunca execução automática: a IA sugere, o operador decide.
  • Versionar método e janela de treino: model_version, train_start, train_end em todo output.
⚠️ ARIMA assume que a série é estacionária. Se o estoque de um SKU está crescendo todo mês (tendência), você precisa diferenciar a série antes de rodar o modelo. Ignorar isso gera previsões completamente erradas.

Redes neurais para forecast — sem hype

Modelo neural só entra quando superar o baseline com estabilidade comprovada
Prioridade 2 — Dados / IA
🧠 Por que isso importa para o Jade-stock

Redes neurais para séries temporais não funcionam como imagens ou texto. Elas precisam de conceitos específicos: janela temporal (lookback), horizonte de previsão, features exógenas. Sem entender esses conceitos, você não vai saber se o modelo está aprendendo o padrão real ou memorizando os dados de treino. E não vai saber quando ele começou a errar — o que em estoque significa reposição errada, falta de produto ou excesso de capital parado.

O que dominar

  • Janela temporal (lookback window): quantos períodos passados o modelo vê para fazer cada previsão. Ex.: usar os últimos 12 meses para prever o próximo mês.
  • Horizonte de previsão: quantos períodos à frente o modelo prevê. Previsão de 1 semana vs 3 meses — modelos diferentes, precisões diferentes.
  • Features exógenas: variáveis externas que influenciam a demanda — feriados, datas comemorativas, campanhas, clima. O modelo aprende com elas, não ignora.
  • N-BEATS, N-HiTS, TFT: arquiteturas modernas para séries temporais. Entenda o que cada uma faz bem antes de escolher.
  • Overfitting: modelo que decora o treino mas erra no mundo real. Como detectar com set de validação e teste separados.
  • Drift: quando o padrão do mundo muda e o modelo fica desatualizado. Como monitorar e recalibrar.
  • Previsão probabilística: em vez de um número único, o modelo retorna um intervalo de confiança. Útil para definir estoque de segurança.

Aplicação no Jade-stock

  • Comparação objetiva: neural vs ARIMA vs baseline ingênuo, documentada com métricas iguais.
  • Registro de model_version, lookback_window, forecast_horizon e métricas em todo output.
  • Fallback automático: se o modelo neural parar de funcionar, o ARIMA assume sem intervenção manual.
  • Features exógenas do Jade-stock: feriados nacionais, datas de inventário, promoções cadastradas.
  • Nenhum modelo escreve diretamente no saldo — apenas emite recomendação para aprovação humana.
⚠️ A janela temporal e o horizonte de previsão são as decisões mais importantes do modelo — e costumam ser as menos discutidas. Uma janela muito curta e o modelo não aprende padrões sazonais. Um horizonte muito longo e a precisão cai drasticamente. Defina ambos com base no ciclo real de reposição do negócio.

MLOps e governança de modelos

Rastreabilidade de experimento, produção e rollback sem improviso
Prioridade 2 — Dados / IA
🧠 Por que isso importa para o Jade-stock

Sem MLOps, você não sabe qual versão do modelo está em produção, com quais dados foi treinado, nem por que a precisão caiu na semana passada. MLOps é o Git para modelos: rastreia tudo, permite reverter, e define critérios claros de quando um modelo pode ir para produção — e quando deve ser removido.

O que dominar

  • Tracking de experimento: registrar cada treino com parâmetros, métricas e artefatos.
  • Versionamento de modelo e dataset: saber exatamente com quais dados cada versão foi treinada.
  • Critério de promoção: o que precisa ser verdade para um modelo ir de "experimento" para "produção".
  • Rollback: como voltar para a versão anterior quando o modelo atual começa a errar mais.
  • Auditoria do output: todo resultado da IA registrado com correlation_id, model_version e timestamp.

Aplicação no Jade-stock

  • Saída da IA sempre com metadados: model_version, train_end, mae, correlation_id.
  • Checklist de promoção: novo modelo só vai para produção se bater o anterior nas últimas 4 semanas de dados reais.
  • Fallback automático: se erro do modelo aumenta acima do threshold, o sistema reverte para a versão anterior.
  • Registro de recomendações no schema ia — separado dos dados de estoque, nunca misturado.

Contábil orientado a eventos

6 sub-módulos progressivos — partidas dobradas até fechamento e reconciliação
Prioridade 3 — Operação
🧠 Por que isso importa para o Jade-stock

O módulo Contábil transforma eventos de estoque em lançamentos financeiros auditáveis. Sem entender contabilidade, você não sabe se o código está fazendo a coisa certa — e um erro silencioso num lançamento contábil pode comprometer o balanço do cliente. Cada sub-módulo abaixo tem um escopo específico e uma entrega concreta. Faça um de cada vez.

📌 Estes 6 sub-módulos devem ser estudados em ordem. Cada um depende do anterior. Não pule para DRE sem entender partidas dobradas.
1
Partidas dobradas e plano de contas Entrega: matriz de mapeamento evento_wms → lançamento contábil
Conceito em 30 segundos

Todo lançamento contábil tem dois lados: débito e crédito, sempre em valor igual. Isso se chama "partidas dobradas". Ex.: quando o WMS registra uma entrada de mercadoria, o contábil debita Estoque (ativo sobe) e credita Contas a Pagar (passivo sobe). O plano de contas é a lista organizada de todas as contas que o negócio usa.

O que dominar

  • Débito vs crédito: o que cada um faz em contas de ativo, passivo, receita e despesa.
  • Natureza das contas: ativo aumenta com débito, passivo aumenta com crédito.
  • Plano de contas: grupos (ativo circulante, passivo, receita operacional, custo) e subcontas.
  • Lançamento manual: como registrar entrada de mercadoria, venda, devolução e avaria.

Entrega concreta

  • Tabela de mapeamento: evento_wms → débito → crédito → histórico para cada tipo de evento do WMS.
  • Ex.: recebimento_conferido → D: Estoque / C: Fornecedores → "Entrada NF #X".
  • Plano de contas mínimo do Jade-stock com os grupos relevantes para WMS.

Demonstrações financeiras — DRE, BP, DFC e DMPL Entrega: estrutura dos reports financeiros automáticos do Jade-stock
Conceito em 30 segundos

A DRE (Demonstração do Resultado) mostra se o negócio lucrou ou não em um período. O BP (Balanço Patrimonial) mostra o que o negócio tem (ativos) e o que deve (passivos) em uma data. A DFC (Demonstração do Fluxo de Caixa) mostra quanto dinheiro entrou e saiu — diferente do lucro, que inclui coisas que ainda não foram pagas. O Jade-stock pode gerar todas essas automaticamente a partir dos eventos.

O que dominar

  • DRE: Receita → CMV → Lucro Bruto → Despesas Operacionais → EBITDA → Resultado Líquido.
  • BP: Ativo Circulante (estoque, caixa, clientes) vs Ativo Permanente vs Passivo vs Patrimônio Líquido.
  • DFC: método indireto (ajustes sobre lucro) vs direto (entradas e saídas de caixa).
  • Diferença entre regime de competência (DRE) e regime de caixa (DFC).

Entrega concreta

  • Estrutura de query SQL que gera DRE por período a partir dos lançamentos da event_store.
  • Mapeamento de contas do Jade-stock para as linhas da DRE (Receita de Vendas, CMV, etc.).
  • Esboço do BP mínimo gerado a partir do saldo das contas contábeis.

Fechamento contábil e reconciliação Entrega: rotina mensal automatizada e relatório de divergências
Conceito em 30 segundos

Fechamento contábil é o processo mensal de "encerrar os livros": verificar que todos os lançamentos estão corretos, gerar as demonstrações do mês e apurar o resultado.   • Script de fechamento: query que gera o balancete mensal a partir dos lançamentos do schema contabil.
  • Consumer contábil com retry + DLQ: nenhum evento crítico perdido sem rastreamento.
  • Relatório de reconciliação: saldo WMS vs saldo Contábil, com divergências explicadas e linkadas ao correlation_id.

H

IAM — OAuth2, OIDC e licença por tenant

Segurança de acesso e modelo comercial sem acoplar lógica no cliente
Prioridade 3 — Operação
🧠 Por que isso importa para o Jade-stock

O Jade-stock já tem OAuth2 planejado via Google. O que você precisa entender é o que acontece depois do login: como o token do Google vira um token do Jade-stock, o que é um scope (permissão específica), como cada tenant (cliente) fica isolado dos outros, e como a licença é validada via orquestrador sem intervenção manual. Sem isso, você não consegue explicar para um cliente por que o operador X não pode ver os dados do operador Y.

O que dominar

  • AuthN vs AuthZ: autenticação (quem é você?) vs autorização (o que você pode fazer?).
  • OAuth2 e OIDC: o fluxo completo de "Entrar com Google" até o token interno do Jade-stock.
  • PKCE: proteção obrigatória para apps clientes (Flutter/Flet). Por que usar e como funciona.
  • Scopes: permissões granulares. Ex.: wms:read, wms:write, contabil:read.
  • Tenant isolation: como garantir que os dados de um cliente nunca vazam para outro.
  • Feature flags por plano: habilitar ou desabilitar módulos (IA, Contábil) por licença ativa.

Aplicação no Jade-stock

  • API valida token + scope em cada endpoint — sem token, sem acesso.
  • Matriz de acesso: operador de estoque vs gerente vs contador — permissões diferentes por perfil.
  • Controle de módulo por licença: se o cliente não pagou pelo módulo de IA, o endpoint retorna 403.
  • Auditoria de acesso: quem acessou o quê, quando e de qual IP — para ações sensíveis como ajustes de estoque.

I

Observabilidade e operação contínua

Saúde do sistema visível em tempo real, com resposta rápida a incidentes
Prioridade 3 — Operação
🧠 Por que isso importa para o Jade-stock

Quando um cliente ligar dizendo que o sistema está lento ou que um evento sumiu, você precisa conseguir responder em minutos — não em horas de investigação no terminal. Observabilidade é a capacidade de entender o que está acontecendo dentro do sistema apenas olhando para seus outputs: logs, métricas e traces. Sem isso, você está voando no escuro em produção.

O que dominar

  • Logs estruturados: JSON com correlation_id, tenant_id, level e timestamp em todos os logs — não texto livre.
  • Métricas: latência (p50, p95, p99), taxa de erro e throughput por endpoint.
  • Tracing distribuído: rastrear uma requisição que passa por WMS → Event Store → Contábil usando o mesmo correlation_id.
  • SLO (Service Level Objective): meta de disponibilidade. Ex.: 99.5% das requisições em menos de 500ms.
  • Alerta acionável: alerta que diz o que fazer, não só que algo está errado.
  • Runbook: documento com o passo a passo de como resolver cada tipo de incidente.

Aplicação no Jade-stock

  • Dashboard mínimo: latência da API, taxa de erro, eventos pendentes na fila e eventos na DLQ.
  • Alerta de DLQ não vazia: se há eventos que falharam 5 vezes, o alerta dispara imediatamente.
  • Trace de operação: dado o correlation_id de um lançamento contábil, encontrar o evento WMS de origem em segundos.
  • Playbook por módulo: o que fazer quando WMS cai, quando Contábil para de processar, quando modelo de IA começa a errar.

termos que aparecem sem explicação no código
ARIMA/SARIMA

Modelos estatísticos para prever séries temporais. SARIMA adiciona componente de sazonalidade.

Outbox Pattern

Salvar evento no banco na mesma transação que o dado, garantindo que nunca se perca na publicação.

Idempotência

Processar o mesmo evento duas vezes produz o mesmo resultado que processar uma vez.

DLQ (Dead-Letter Queue)

Fila de destino para eventos que falharam muitas vezes. Nada é descartado silenciosamente.

MLOps

Práticas de engenharia para operar modelos de ML com rastreabilidade, versionamento e rollback.

Tenant

Um cliente/organização com dados completamente isolados dos outros no mesmo sistema.

OIDC

OpenID Connect: camada de identidade sobre OAuth2. É o que o "Entrar com Google" usa.

SLO

Service Level Objective: meta técnica de qualidade. Ex.: 99.5% das requisições em menos de 500ms.

Stationarity

Série temporal sem tendência ou sazonalidade. Exigida pelo ARIMA antes de rodar.

ACF / PACF

Gráficos de autocorrelação que revelam os parâmetros p e q do ARIMA.

Lookback window

Quantos períodos passados o modelo neural usa para fazer cada previsão.

Feature exógena

Variável externa que influencia a série (feriado, clima, promoção) — o modelo aprende com ela.

Drift

Quando o padrão do mundo muda e o modelo que era bom começa a errar. Exige recalibração.

CMV

Custo das Mercadorias Vendidas. O custo real do produto: compra + frete + perdas + armazenagem.

Partidas dobradas

Todo lançamento tem dois lados iguais: débito e crédito. A base de toda contabilidade.

PKCE

Extensão de segurança para OAuth2 em apps clientes. Evita interceptação do token.
